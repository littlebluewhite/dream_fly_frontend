# 課程報名與方案訂閱:獨立、皆為模擬付款

兩種付費產品——「課程」(結帳後產生**報名**)與「方案／月票」(結帳後產生**訂閱／使用權**)——各自獨立結算,且**彼此不互相涵蓋**:持有任何方案(含無限會員卡)都不會自動折抵或免除任一課程的報名費用。兩者都只產生**模擬**的線上付款成功(無真實金流、無自動續訂、不寫入真實排程)。

## 背景與決定

auth-at-checkout 將訪客購物車、登入閘與結帳收斂到單一持久化購物車(以會員 `cart` 為基準)。結帳時依品項 `type` 分流:

- `type: 'course'` → 走「報名」:回饋點數的 mock,**不**寫入 `MY_COURSES`／日程(既有「課程已加入你的日程」純為展示文案)。
- `type: 'pass'` → 走「訂閱」:寫入**持久化**的 `subscriptions` 使用權(localStorage,跨 reload／登入存活)。

選擇兩者獨立,是為反映領域語意:方案是「入場／上課資格」,報名是「加入某一具體班別名單」,概念不同(見 `CONTEXT.md`)。

## 後果(刻意,非 bug)

- **持月票仍需逐堂付費報名**:一名持有「無限會員卡」的會員,在系統中仍會被要求為每一堂課程付費報名。這是**刻意延後**的範圍決定——「方案涵蓋課程報名」需要一套涵蓋／扣抵規則,留待日後。若未來看到此行為而疑似 bug,請先回看本 ADR。
- **真實金流、自動續訂、寫入真實排程**:皆延後。

## 已知後續(codex 對抗式審 round 3,P2,延後)

持久化 client store(`authStore`、會員 `cart`、`subscriptions`)在 SSR 下於 module-init 即讀 localStorage,於「已登入／有訂閱」狀態硬重整 SSR 頁面時,可能造成 hydration 首屏閃動或警告。此沿用既有 `src/lib/stores/cartStore.ts` 的相同 load-at-init 模式;正解為「持久化 store 一律延後到 mount 後再讀」,且應連同既有 cartStore 一併統一處理,故不在本 PR 範圍,另案跟進。
